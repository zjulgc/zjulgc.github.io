Studying the Use of Java Logging Utilities in the Wild

日志工具分析 实证研究 ICSE 2020

RQ1:What are the external LUsbeing used in the wild?

There are four main reasons behind the use of ELUs: (1) General-purpose logging, (2) LU interactions, (3) Internationaliza- tion, and (4) Modularization. Majority of the projects use ELU for general-purpose logging. 83 out of the top-100 mostly used ELUs are used for LUinteractions. Some ELUs provide unique features like internationalization and modularization, but they are only used by a very small number of projects.

RQ2:Why do developers implement ILUs in their projects?

There are three project specific needs, which lead to the implementation of ILUs: (1) defining the logging format, (2) compatibility with other LUs, and (3) ease of configuration and dependency management. Among these three needs, defining the logging format is the most common one. 55.3% of the sampled projects implemented ILUs for this specific need. As the size of the projects grow larger, more projects implement ILUs to satisfy the needs of compatability with other LUs.

RQ3: How are multiple LUs used in the wild?

There are four usage contexts behind the use of multi- ple LUs in one project: (1) interaction with LUs from the imported packages; (2) managing the logging contents; (3) formatting log- ging messages across different components; and (4) developer convenience. Except for developer convenience, the percentage of these usage contexts increases as the project sizes increase.

Exposing Library API Misuses via Mutation Analysis

API误用问题 ICSE 2019

Overview

现状

  • 现有检测API误用的工具基本都是从大量正确使用API的样例中挖掘正确的使用模式，而与这些经常使用的方式不同的就是API误用，但是这些正确的样例是非常难以获得的。另一方面由于假设过于简单，这些工具也不够准确。

本文工作

  • 使用突变分析的两个基础：

    • API误用其实是正确使用的突变
    • 通过执行测试套件能够区分出突变体的误用

  • 本文首先设计了8种突变操作，误用的形式通过分析被杀死的变体来得到

  • 本文能够通过分别在正确以及突变体执行测试组件以及分析执行信息来判断这个突变体是否产生了API误用

主要贡献

  • 如何模拟生成API误用是关键，MUTAPI models correct API usages as structured call sequences based on a predefined grammar.
  • 如何保证被杀死的变体中有API误用也是一个挑战，因为一个变体被杀死的原因有很多，不一定是因为API误用，本文通过分析 the failing stack traces of the killing tests，分析过程：
    • 通过分析栈找到toot cause，如果root cause是库API，那么就极有可能是库API有问题
    • Second, killing stack traces of a target API should be specific to this API. If a killing stack trace is also observed in the mutation analysis of other APIs, it is less likely to be caused by the target API’s misuses.
    • Third, killing stack traces of a target API should not be specific to a certain usage. If a killing stack trace is only observed in the mutation analysis of a specific project, it is more likely to be caused by the bugs specific to this project instead of a general misuse of the target API. MUTAPI
  • 识别出API后，如何有效的提炼出API误用的模式？it first models an API misuse as a pair of a violation type and an API usage element. It then selects those most frequently observed violation pairs as API misuse patterns.

Conducting Mutation Testing

An Analysis and Survey of the Development of Mutation Testing

突变测试综述 TSE 2011

两个假设

  • the Competent Programmer Hypothesis (CPH) 有能力的程序员也会犯错
  • the Coupling Effect 大的错误是小的错误组成

分析流程

Does mutation testing improve testing practices?

通过分析突变体数据集证明突变分析可以提高测试质量

本文要解决的四个问题

对测试数量的影响

  • 使用变异测试会使得开发人员编写更多的测试用例

对测试质量的影响

  • 测试套件需要提升以杀死更多的变体，在提高的过程会超出当时编写的目标

故障耦合

  • 如果将突变测试用于引入故障的更改，它会报告一个活的突变体，该突变体被错误修复更改中的测试杀死

变体冗余

  • 所生成的所有突变体不会都被执行，只有一部分样本会被执行，每行代码生成的突变体不会超过一个；根据某行突变的一个变体被杀死，那剩下的变体也很可能会被杀死

摘要

  • 代码覆盖率可视化工具已经有商业应用[Tool](Code coverage at google-ESEC/FSE)

  • 代码覆盖率的足够阈值本质上是任意的，对于代码覆盖率的有效性是有争议的

  • 突变测试能够解决的问题：

    • 人工变体能够评估测试套件
    • 幸存突变构成了测试的具体目标，能够指出改进测试套件的可能方法

  • 大量突变体限制了使用突变测试的场景，大规模使用突变测试的论文[1](Practical mutation testing at scale)[2](State of mutation testing at Google)

结论

  • 总而言之，本文通过研究谷歌的编译测试数据集，证实了变异测试的假设，且说明了变异测试能预防错误的引入

Practical Mutation Testing at Scale

摘要

  • 现在的突变测试测试技术生成了大量的突变体，导致需要大量的人力和物力，这使得使用突变测试几乎不可能
  • 本文的主要贡献(在不改变效果的同时，尽可能减少突变体的质量)：
    • 变异测试是增量式完成的，在代码审查期间只变异更改的代码，而不是整个代码库；
    • 过滤突变体，去除可能与开发人员无关的突变体，并限制每行和每个代码审查过程的突变体数量；
    • 根据变异算子的历史表现选择变异体，进一步剔除无关变异体，提高变异体质量。

MuSim: Mutation-based Fault Localization Using Test Case Proximity

摘要

  • 错误定位技术可以使用通过和失败的测试用例中收集的信息来定位错误的语句。本文提出的MuSim根据计算出的与不同突变体的相似度来识别错误语句。

Intro.

错误定位技术中通常使用程序语法、语义和程序元素的运行时行为以及测试用例执行结果中存在的信息进行错位定位技术

此外还有程序切片：

  • 动态切片
  • 混合切片
  • 关键切片
  • 基于频谱的故障定位技术
  • 机器学习技术
    • SVM
    • 决策树
    • 集成分类器

Though NN-based techniques are popular, they require considerable time for training and require estimation of several parameters.

  • 本文将新系统执行错误与变体的执行错误相比较来定位错误语句

Overview

总结

  • 本文所述工作是将程序执行信息与突变体错误信息进行相似匹配，来获得与产生错误信息想进的突变体

  • 使用突变测试来进行错误的问题：

    • 首先是突变操作，难以保证突变操作能够包含所有的真实错误操作，因此只能采取尽可能多的突变操作或得尽可能多的突变体
