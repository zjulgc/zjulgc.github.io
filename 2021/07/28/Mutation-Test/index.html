突变测试

An Analysis and Survey of the Development of Mutation Testing

TSE 2011(综述)

什么是突变测试？

  • 一种基于故障的软件测试技术，突变测试通过突变来模拟程序员的错误

突变测试两个假设

the Competent Programmer Hypothesis (CPH)

  • 意义：程序员行为角度的分析
  • 含义：程序员是有能力的，其所开发的程序是接近正确版本的
  • 这个假设阐述了也许在程序员交付产品的时候，产品存在一些错误，但是这些错误只是一些简单的错误，可以通过修正一些语法错误来修正错误。因此在突变测试中，通过构建语法错误来构建的变代表了“有能力的程序员”所犯的错误。

the Coupling Effect(耦合效应)

  • 意义：耦合效应阐述了突变分析中的错误类型，通过这个假设，在突变分析中可以去除高阶突变体

  • 含义：检测程序中所有简单故障中的测试数据能够检测出大部分复杂故障。

  • 这个假设可以简单的理解为，复杂错误是多个简单的错误组成的，因此我们只需要找到简单错误即可，这个假设主要在消除突变体中起作用。

突变分析过程

输入

  • 原始程序P
  • 通过突变P得到的突变体集合P’
  • 测试数据T

运行过程

杀死：在P’上运行的结果与在P上运行得到的结果不同

Equivalent Mutants：表示某些没有被杀死的突变体可以通过增强测试数据杀死，但是某些测试数据无论怎样修改测试数据都不能被杀死，这些突变体就是突变分析不能被广泛使用的一个障碍

  • 将T中所有测试数据在P上运行：
    • P运行正确：
      • 将T中所有测试数据运行在P’上：
        • 所有突变体被杀死：结束
        • 部分突变体没被杀死：分析并标记Equivalent Mutants
    • P运行出错：
      • 修改程序P并重新开始流程

Mutation Score

  • 这个分数展示了输入的测试数据的质量

一些突变规则

一些问题

  • 测试数据在大量突变体上执行需要花费大量的时间
  • 另一些问题与突变测试投入的人力有关，例如the human oracle problem and the equivalent mutant problem

总结

总而言之，突变测试是一种计算成本非常高的测试技术，因此现有的工作主要在解决三个问题：如何做的更少，如何做的更快，如何做的更聪明，即减少生成的突变体以及降低执行成本。

The Unit Test Quality of Deep Learning Libraries: A Mutation Analysis

Main Idea

  • 本文主要是对深度学习库的单元测试质量进行实证研究，通过突变出一定数量的错误并注入到深度学习库中，再使用原有的测试用例进行测试，以此检测深度学习库中单元测试的质量。

本文贡献

  • RQ1：如果将突变体注入被测系统 (SUT)，可以在多大程度上杀死突变体？
    • 60%的突变体被杀死，现有的测试用例在检测错误逻辑流引起的错误方面比错误值引起的错误更有效（发现 1），但它们无法检测到装饰语句上的突变（发现 2）。 在被杀死的突变体中，超过 60% 的突变体没有被测试用例中的断言检测到，而是被异常揭示（发现 3）。
  • RQ2：如果将突变体注射到依赖成分 (DOC) 中，可以在多大程度上杀死突变体？
    • 存活的突变体从大约 30% 的 SUT 增加到超过 50% 的 DOC
  • RQ3：哪些测试用例可以有效杀死突变体？
    • 发现 5 表明 Theano 可能没有完全使用单元框架的断言，测试用例中的断言比其他文件中的断言更有效地杀死突变体（发现 6）。当通过抛出的异常揭示突变体时，它们中的大多数都会抛出 TypeError、ValueError 和 IndexError（发现 7）。 与 Python 定义的异常相比，很少使用自定义异常，并且很少通过自定义异常揭示突变（发现 8）。 对于 Theano 和 TensorFlow，预处理阶段的突变体比其他阶段更难被杀死（发现 10），并且一些自定义异常与特定的深度学习阶段相关（发现 11）。
