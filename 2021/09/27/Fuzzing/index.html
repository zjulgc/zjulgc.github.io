The Progress, Challenges, and Perspectives of Directed Greybox Fuzzing

本文是导向型模糊测试的综述

Intro.

  • 大多数灰盒模糊测试是基于覆盖率引导的，目的是在有限的时间预算内覆盖尽可能多的程序路径

  • 但是将不同的代码视为相同的是不合适的，有些代码的bug可能性会更大，通过盲目地扩展代码覆盖范围来测试软件效率较低

导向型模糊测试

  • 导向型模糊测试会将大部分时间花在特定目标位置(容易出现bug的地方)

    导向型测试的三个阶段

    • 传统的导向型模糊测试基于符号执行(出现复杂的约束求解和路径爆炸的问题)
    • AFLGo:手动标记目标和基于距离（种子与目标）的导向型灰盒模糊测试
    • 在不同的场景下，会使用不同形式的目标（如目标序列）来使用不同的方式（如序列覆盖率）引导Fuzzing

背景

AFL

原理

AFL是基于覆盖引导的灰盒模糊测试技术中的代表性工作，以下是AFL的原理

  • AFL在程序中插桩以获得边覆盖率
  • 从种子队列中选择一个种子对其赋予能量并进行突变
  • 如果突变的种子触发了新的路径或者是触发bug都会将其作为一个新的种子添加到种子队列中

CGF核心技术可以大致分为覆盖粒度，突变策略，种子优先级，能量调度

● 覆盖粒度：常使用的覆盖粒度有块覆盖，边覆盖（也可细分为上下文敏感和上下文不敏感）。AFL的覆盖粒度选择的是上下文不敏感的边覆盖策略。它在编译时为每个分支跳转插入随机数，并在运行时从寄存器中收集这些插入的数字，以识别基本块转换。边缘覆盖考虑了基本块之间的过渡，比基本块覆盖更加精细敏感。它比路径覆盖更具可伸缩性，因为它避免了路径爆炸。

● 突变策略：AFL有两类突变策略:确定性策略和非确定性策略。首先应用确定性策略，它利用基于位翻转、算术、令牌、字典和兴趣值的变异器以不同粒度顺序地变异种子。在执行了确定性策略之后，AFL引入了非确定性策略，包括havoc阶段和splice阶段。在havoc阶段，AFL从确定性策略中随机选择一系列突变操作符，并将它们应用到种子文件中的随机位置，从而对种子进行突变。因此，生成的testcase与原始种子有很大的不同。在splice阶段，AFl会从种子队列中随机选择另一颗种子，并将其与当前的种子重新组合，生成新的种子。然后，对新种子重新实现havoc策略。（通常是在很久未触发新路径或者bug时才会执行不确定策略）
● 种子优先级：AFl会根据种子运行的边覆盖率和触发bug的数目来确定其优先级。
● 能量调度：AFL会根据种子的运行时间，种子的大小以及种子边覆盖率和触发的bug来给与种子不同的能量。（能量指的是该种子可以突变的数量）

工程

执行过程

①从源码编译程序时进行插桩，以记录代码覆盖率（Code Coverage）；
②选择一些输入文件，作为初始测试集加入输入队列（queue）；
③将队列中的文件按一定的策略进行“突变”；
④如果经过变异文件更新了覆盖范围，则将其保留添加到队列中;
⑤上述过程会一直循环进行，期间触发了crash的文件会被记录下来。

构建语料库

  • AFL需要一些初始输入数据（也叫种子文件）作为Fuzzing的起点，这些输入甚至可以是毫无意义的数据，AFL可以通过启发式算法自动确定文件格式结构。
  • 如果要获得更快的Fuzzing速度，那么就有必要生成一个高质量的语料库

    • 选择输入文件：有效输入，尽量小的体积(AFL建议是小于1KB)

    • 寻找方法：使用项目自身提供的测试用例；目标程序bug提交页面；使用格式转换器，用从现有的文件格式生成一些不容易找到的文件格式；afl源码的testcases目录下提供了一些测试用例；其他大型的语料库

      afl generated image test sets

      fuzzer-test-suite
      libav samples
      ffmpeg samples
      fuzzdata
      moonshine

    • 语料库蒸馏(精简大型语料库)-afl-cmin afl-tmin

      afl-cmin (移除执行相同代码的输入文件)：尝试找到与语料库全集具有相同覆盖范围的最小子集

      afl-tmin (减小单个输入文件的大小)：两种工作模式instrumented mode crash mode

    • 这两种模式可以迭代使用

构建被测试程序

  • afl-gcc / afl-g++ 是 gcc / g++的装饰器，前者输入的参数会传递给后者，在编译的过程中，只需要将编译器设置为afl-gcc / afl-g++ 即可
  • llvm 模式

开始fuzzing

Directed Greybox Fuzzing

  • DGF会将更关注于目标点的覆盖，而不是全代码的覆盖。AFLGo是第一个导向型的灰盒模糊测试技术，接下来以AFLGo来说明DGF的原理。首先AFLGo会手动的设置目标点和计算目标基本块到其他基本块的距离。然后在模糊测试中，会更具覆盖率和距离来判断该输入是否添加到种子队列。并且会根据模拟退火策略基于种子不同的能量。（与AFL的核心区别，就是AFLGo会利用距离特性逐渐让种子执行路径靠近目标点）

Difference between CGF and DGF

● Seed prioritization ：由于CGF的目标是最大化路径覆盖，CGF优先选择触发新路径的种子。不同的是，DGF的目标是到达代码中的特定位置。因此，它会优先选择那些“更接近”目标的种子。种子的评估指标变化很大，包括距离、覆盖、路径和概率。
● Target involvement ：DGF会对程序进行设置目标，从而使得DGF比CGF更有方向性，更可能触发bug。（并且我们利用目标之间的关系来加速检测复杂的行为bug）
● Exploration-exploitation：DGF有两个阶段，分别为exploration 和 exploitation，前者目的是使得fuzzing可以获得更多的信息，可能有更多的突变的种子，后者的目的只是让种子接近目标。然而CGF没有。

几个方向

  • Patch testing：DGF可以用来测试补丁是否完整和兼容。

  • Bug reproduction：DGF在没有错误输入的情况下再现已知错误时非常有用。

  • Knowledge boost ：DGF可以通过集成来自人类分析人员或辅助技术的知识来促进程序测试。

  • Energy saving：DGF可以节省花费在代码区域等非bug区域上的时间和计算资源，识别关键代码区域来指导测试比以无导向的方式测试整个程序更有效。

  • Special bug hunting：DGF可以应用于基于自定义指标的特殊bug搜索。

一些挑战

二进制fuzzing的问题

目前很多的DGF对源代码依赖性比较强，如果针对于二进制程序，很多的DGF都不可用。

  • 繁重的运行时开销。因为简单的二进制代码模糊测试通常需要利用一个全系统仿真器，而目前的全系统仿真器通常效率比较低，开销比较大。
  • 目标信息收集困难。对于开源的PUT，我们可以从各种渠道获得目标信息，比如CVE漏洞描述、git提交日志中的更改，以及源代码中关键位置的人工体验。但是，对于一个放入二进制代码，我们只能从bug跟踪中提取目标信息。
  • 难以确定目标。源码可以很好的确定目标，但是对于二进制而言，确定目标就很困难，可能需要利用IDA pro对其进行反汇编。

可以通过硬件辅助解决

针对二进制码级目标识别标注问题，可以利用基于机器学习的方法或启发式二进制扩散方法自动识别脆弱码。

目标的标定

大多数已知的导向模糊测试工具需要人工手动的标记目标

  • 我们可以使用静态分析工具找到PUT中潜在的危险区域（这些工具通常是特定于所使用的bug类型和编程语言的）
  • 或者可以利用compiler sanitizer passes （例如UBSan）注释PUT中的潜在bug。
  • 基于深度学习的方法对脆弱性进行预测
  • 可以通过基于Bindiff的二值级比较，提取不同的函数及其不同的基本块，从而识别目标的补丁分支。

优先级指标的确定

例如基于距离的种子优先级，只考虑分支的长度，没有考虑分支跳转之间的概率，所以这种不精确性会限制DGF的性能

  • 一个可能的解决方案是考虑分支跳转概率。当基于概率评估目标的可达性时，每个种子都是根据种子生成一个输入到达目标的可能性来排序的，也就是将该种子的当前执行路径转换为经过目标的目标路径的概率。由于执行路径可以看作是连续分支的马尔可夫链，因此路径的概率可以通过收集路径内所有分支的概率来计算。利用蒙特卡罗方法统计计算出分支概率的比值，从而估计出分支概率。 这种基于概率的方法一个可能的缺点是潜在的运行时开销。统计跳跃计数和概率计算都引入了额外的计算。缓解性能下降的一个简单方法是区间采样。另一种可能的解决方案是加速计算，这涉及到如何存储和访问元数据（其实就是设计一种新型的数据结构）。

全局优化偏差

当DGF测试中存在多个目标时，如何协调这些目标是另一个挑战。

  • 一种策略是基于Dijkstras算法寻找全局最短距离，就像AFLGo一样。但是，这种全局最优可能会错过最接近某个目标的局部最优种子，从而导致偏差。另一种协调多目标的策略是目标分离。对于每个种子，只选择所有目标之间的最小距离作为种子的距离，并根据最小距离对种子进行优先排序。这样，我们可以避免局部最优偏差，但这可能会减慢达到特定目标的速度。

Seed Selection for Successful Fuzzing

摘要

灰盒模糊测试会输入种子，然后启动fuzzing

  • 在评估一个fuzzer的时候，我们需要构建一个语料库，有几种方法：
    • 使用空文件
    • 使用目标输入格式的单个种子
    • 收集大量的种子
  • 问题：没有人评估不同的选择究竟会对fuzzer的结果影响是否相同，也没有人评估哪种方法的效果更好
  • 本文的贡献：评估了种子的选择对fuzzer发现错误的能力的影响，且发现模糊测试的结果收到初始化种子的影响，最小化语料库效果最好
    • 调研了实际使用中种子的选择策略以及最小化语料库技术，选择种子的方法千差万别，并且没有达成共识
    • 一种新的语料库最小化工具OptiMin
    • 使用以后的最小化语料库工具与OptiMin 进行定量的对比

相关背景

  • fuzzing生成测试用例的方法有两种：突变和生成
    • 基于生成的fuzzer需要规范化输入的格式
    • 基于变异的fuzzer需要一个种子的语料库开始初始化用例的生成，然后逐渐变异

种子的选择

过去模糊评估的总结，重点是种子选择，采用 Klees 等人使用的类别和符号。 R表示随机抽样的种子； M表示人工构建的种子； G 表示自动生成的种子； N 表示有效性未知的非空种子； V 表示论文假设存在有效种子，但出处未知； E 表示空种子； / 表示在不同的程序中使用了不同的种子，但在一个程序中只使用一种种子； 空白单元格表示该论文的评估未提及种子选择。 我们引入了一个额外的类别：V* 表示具有已知出处的有效种子。 我们还指出评估是否可重复使用相同的种子。

  • 使用不同的语料库评估选择种子的重要性，实验结果如下，总体来说实验结果表明使用有一定关联的种子会提高覆盖率

  • 较大的语料库拥有更多的种子，但是有很大的缺陷，会阻塞突变队列，总成了很多冗余
