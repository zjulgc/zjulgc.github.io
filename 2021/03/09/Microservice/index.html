前置知识

微服务

定义

微服务架构风格是一种将一个单一应用程序开发为一组小型服务的方法，每个服务运行在自己的进程中，服务间通信采用轻量级通信机制(通常用HTTP资源API)。这些服务围绕业务能力构建并且可通过全自动部署机制独立部署。这些服务共用一个最小型的集中式的管理，服务可用不同的语言开发，使用不同的数据存储技术。

优势

开发角度

  • 不同服务的开发人员可以选择自己擅长的技术栈进行自己负责的服务的开发，而不用统一开发语言等
  • 各个服务可以独立的更新修改等

软件系统的角度

  • 可拓展性：微服务中每个服务都是高聚的，也就是功能比较单一，这样方便我们按照新的需求选择相应的模块进行拓展
  • 高生产力：开发不同服务的团队可以并行开发
  • 可重用性：已有的服务可以做为新服务的轮子，如果某个服务过时，我们对齐进行重构再上线

挑战

  • 通信复杂：每个服务之间的通信需要保证通信的准确性
  • 维护成本增加：每个服务都有独立的数据库，每个服务都需要独立的运维
  • 测试更加复杂：数据库和其他服务的问题有可能导致测试的失败

预备知识

数据并行作业的DAG

DAG不仅能够作为分布式系统的运算模型，在单体应用中，随着业务逻辑的增加，也会出现不同变量之间的出现依赖，这也需要根据业务的依赖关系确定正确的执行顺序。由于本文是在对比微服务与分布式运算模式，所以这里将主要阐述分布式系统的运算模式。

  • 举个例子来说，如上图，每个圆表示一个任务，有向边表示任务之间的依赖关系，在图中2指向6表示、任务6依赖于任务2，这就表示如果我们要完成任务6时，要先执行任务2。
  • 因此，我们如果要执行某个任务的时候，要根据这个DAG生成任务的执行路径

重尾分布

重尾分布是一种随机变量的分布，主要表现为少量个体做出大量贡献，一个比较著名的对应就是二八定律(80%的结果取决于20%的原因)

  • 指数分布在 x→∞ 的时候是以指数的速度趋近于0，那么以指数分布为分界线，我们把 x→∞ 时下降速度更快的称为 Thin-tailed distribution （好像还没有中文翻译），比如正态分布。也就是说，在远离峰值的尾部区域，事件发生的概率更低一些。所以正态分布用来对那些主流事件发生较多，非主流事件发生较少的情况进行建模更为合适。

  • 与此相对的，把 x→∞ 时下降速度慢于指数分布的成为重尾分布（Heavy-tailed distribution），数学定义为：

    lim⁡x→∞eλxFˉ(x)=∞,for all λ>0\lim_{x \to \infty} e^{\lambda x}\bar{F}(x) = \infty,for \, all \, \lambda > 0 x→∞lim​eλxFˉ(x)=∞,forallλ>0

重尾分布更适用于对那些离峰值较远的稀有事件也会有相当的概率发生的情况。重尾分布作为一个大的类别，还包含三个重要的子类别，分别是肥尾分布（Fat-tailed distribution），长尾分布（Long-tailed distribution）和次指数分布（Subexponential distribution）。

  • 长尾分布：

    长尾分布，或者说长尾理论是一个与互联网发展分不开的概念。说到这里就不得不先提一下传统商业中的帕累托法则（Pareto principle），又称为二八定律。比如80%的财富集中在20%的人手里，图书馆里20%的书可以满足80%的顾客。于是大家往往只关注在PDF图中最左面的20%的顾客，以期满足80%，如下图绿色的部分，来实现效益的最大化。

摘要

  • 本文研究了微服务依赖性的特征及其运行时的性能，对比了数据并行作业的传统DAG
  • 几个结论：
    • 微服务调用图是heavy-tail分布，其拓扑与树相似，许多微服务是热点
    • 找到了三种有意义的调用图可以优化微服务设计
    • 大多数微服务对于CPU的干扰敏感的多，相对于内存的干扰
    • 建立了一个数学模型模拟调用图，能够保留图形属性

背景

微服务与数据并行作业的DAG之间的区别

  • 调用图的大小呈现重尾分布

    这个实际上说明生产环境中的微服务调用图大部分是比较小的，但是小部分的调用图非常巨大

  • 调用图拓扑上类似一棵树，大多数节点的入度为1。相比之下，DAG包含多个gather components

  • 5%的微服务被集群中90%的在线服务复用，传统的DAG不共享节点

    微服务&k8s时动态调用过程，尤其是在某个服务负载不够时?

  • 微服务运行时出现高度动态的调用关系，一个在线服务用9类以上的拓扑学上不同的图，相比之下，DAG往往是静态的

微服务的强依赖性为微服务的设计提供了良好的机会

  • 这里说的强依赖主要是两个服务各自接口之间的依赖性，也就是真的会导致无限调用的那种强依赖。本文提到的优化就是对导致这种微服务强依赖性的两个接口进行耦合，以防出现bug。

微服务的性能对CPU扰动比对内存干扰要敏感

这里说成是微服务的性能或许不太准确，可能根本上受到扰动的是docker或者k8s的节点

  • 微服务与批处理应用共同存在与多个物理机上
    • 微服务容器的资源里利用率可以低至10%(资源指的是什么？)
    • 响应时间在不同的工作负载上变化不大
  • 30%的CPU利用率会使得响应时间减低20%，相比于10%的利用率，因此需要一个更好平衡CPU利用率的作业调度器
  • 微服务的响应时间与拓扑图相关(这个好像是显然的？图上的路径比较长，那么执行时间一般是要大于路径短的？)

随意模型可以很好的模拟动态微服务调用图

  • 已有的微服务基准都比较小，因此不会出现由于负载增加导致的拓扑结构改变

贡献

  • 研究了大规模微服务调用图的结构属性以及依赖性
  • 描述了微服务运行时的性能，可以作为调度的准则
  • 建立了一个能够生成大规模微服务轨迹的模型，且保留了阿里的轨迹图中的图属性

阿里微服务的运行轨迹

微服务与批处理任务可以位于同一个物理机上，但是会在不同的pod上

数据库以及memcached有状态的微服务会处于一个专门的集群上，不与无状态的以及批处理共享

微服务调用图的分析

基本统计信息分析

  • 微服务调用图的大小是重尾分布的

  • 超过40个微服务的调用图，大约50%是memcached

  • 常见图的深度为3，比DAG关键路径要短(深度大，配置容器难？)

  • 微服务调用图呈现出来比较像一棵树，许多只包含一个长链

  • 许多无状态微服务是热点，这对于开发人员来说在资源分配的时候基本只要着重考虑这些微服务即可

  • 调用图高度动态，后续分析时，需要对调用图进行聚类

图学习算法

  • 编码：每个调用图编码为一个向量(InfoGraph)
  • 使用kmeans聚类，将调用图分类
  • 检验聚类的结果，核方法(graph kernel)

剖析

  • 关于调用图上的微服务是否会继续调用下游服务基本是跟层数有关的
  • 调用的较深时，较为容易出现数据不命中，这就要从数据库中读取，MQ可以减少端到端的RT

无状态微服务的依赖性

  • 循环依赖还是存在的，强依赖在rpc中比较多，最好耦合强依赖，避免死锁

耦合依赖性

  • 先前仅仅通过考虑层数来指导调用的生成，没有考虑实际中出现上层对下层多次调用，所以加入时间指标
  • 合并经常被同时调用的不同微服务之间的不同接口

性能分析

MCR是每个容器在每分钟内收到的微服务呼叫的数量

  • MCR与CPU利用率与Young GC高度相关，与内存利用率无关，所以CPU利用率和Young GC可以反映微服务容器运行压力

  • 分类后的微服务在类内的RT与类外的RT有一些差别，所以拓扑结构对RT有影响

  • CPU利用率高，RT性能降低

Microservices Migration in Industry: Intentions, Strategies, and Challenges

ICSME 2019

Main Idea

本文主要是对微服务的实际案例进行实证研究，研究了微服务迁移的意图，策略和挑战

背景

  • 目前业界的研究主要集中在迁移过程的各个阶段的活动，本文主要是聚焦在迁移的更高层面的意图和策略

  • interview的公司及系统：

实证研究结果

RQ1：将现有的系统迁移到微服务系统的意图是什么？

  • 替换现有系统的原因：种种症状可以归结为缺乏可维护性

    • 概述丢失（S1，S10）
    • 更改成本太高（S4）或容易引起副作用（S11）
    • 可分析性差（S11）
    • 此外，被调查的系统还存在可操作性问题
      • 缺少可追溯性 (S11)
      • 启动时间长 (S9)
      • 更新期间的停机时间 (S6) 或一般难以应用更新 (S14)。
    • 内存消耗和带宽瓶颈在内的性能方面是三个系统（S6、S9、S14）的问题
    • 一些参与者表示，由于技术已弃用（S6、S11）或无法通过设计在现有解决方案中实现（S3、S7、S12），当前的功能要求无法解决
    • 新功能的上市时间是几个系统（S6、S9、S12、S13）的驱动力
    • S6 是一个特例，对其进行重大更改可能需要长达六个月的时间才能投入生产

  • 选择微服务系统的原因：

    • 架构（S2、S5、S6、S7、S9、S12、S14）的可拓展性
    • 开发团队（S1、S3、S6、S7）的可拓展性
    • S4、S5 和 S11 明确提到了针对更小、更好的可管理和可维护单元的目标
    • 公司范围内的战略规定实施微服务（S3、S9、S10）“它很时髦，所以他们就这么做了”（P3）
    • 实现云就绪是两个系统（S9、S13）以及质量属性互操作性（S4、S6）和可靠性（S6、S13）的驱动力
    • 由于 14 个系统中只有 3 个完成了迁移活动（见表 II），因此只获得了部分结果；P1、P7 和 P16 证实了更快的决策和更短的上市时间；P4 的缺陷数量有所减少，而 P2 对系统的长期可维护性充满信心。

RQ2：公司应用了哪些微服务迁移策略和分解方法？

  • 普遍的迁移策略就是重写系统
  • 将复杂的系统进行拆分过于复杂
  • 在构建微服务系统的时候，确定微服务的边界是非常难以解决的问题，某些系统使用领域驱动的模式解决，其他大多数系统都是架构师根据开发经验进行边界的确定
  • CI/CD的自动化并不完全
  • 构建具有容错能力的弹性架构非常重要

RQ3：微服务迁移过程中的主要技术和组织挑战是什么？

  • 成员的协作在瀑布式专项敏捷开发的时候会产生摩擦
  • 架构的转变意味着团队的转变，开发人员会担心自己在新的开发团队中的所处位置
