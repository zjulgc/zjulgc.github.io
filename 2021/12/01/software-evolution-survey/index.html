微服务

Microservices Migration in Industry: Intentions, Strategies, and Challenges

ICSME 2019

Main Idea

本文主要是对微服务的实际案例进行实证研究，研究了微服务迁移的意图，策略和挑战

背景

  • 目前业界的研究主要集中在迁移过程的各个阶段的活动，本文主要是聚焦在迁移的更高层面的意图和策略

  • interview的公司及系统：

实证研究结果

RQ1：将现有的系统迁移到微服务系统的意图是什么？

  • 替换现有系统的原因：种种症状可以归结为缺乏可维护性

    • 概述丢失（S1，S10）
    • 更改成本太高（S4）或容易引起副作用（S11）
    • 可分析性差（S11）
    • 此外，被调查的系统还存在可操作性问题
    • 缺少可追溯性 (S11)
    • 启动时间长 (S9)
    • 更新期间的停机时间 (S6) 或一般难以应用更新 (S14)。
    • 内存消耗和带宽瓶颈在内的性能方面是三个系统（S6、S9、S14）的问题
    • 一些参与者表示，由于技术已弃用（S6、S11）或无法通过设计在现有解决方案中实现（S3、S7、S12），当前的功能要求无法解决
    • 新功能的上市时间是几个系统（S6、S9、S12、S13）的驱动力
    • S6 是一个特例，对其进行重大更改可能需要长达六个月的时间才能投入生产

  • 选择微服务系统的原因：

    • 架构（S2、S5、S6、S7、S9、S12、S14）的可拓展性
    • 开发团队（S1、S3、S6、S7）的可拓展性
    • S4、S5 和 S11 明确提到了针对更小、更好的可管理和可维护单元的目标
    • 公司范围内的战略规定实施微服务（S3、S9、S10）“它很时髦，所以他们就这么做了”（P3）
    • 实现云就绪是两个系统（S9、S13）以及质量属性互操作性（S4、S6）和可靠性（S6、S13）的驱动力
    • 由于 14 个系统中只有 3 个完成了迁移活动（见表 II），因此只获得了部分结果；P1、P7 和 P16 证实了更快的决策和更短的上市时间；P4 的缺陷数量有所减少，而 P2 对系统的长期可维护性充满信心。

RQ2：公司应用了哪些微服务迁移策略和分解方法？

  • 普遍的迁移策略就是重写系统
  • 将复杂的系统进行拆分过于复杂
  • 在构建微服务系统的时候，确定微服务的边界是非常难以解决的问题，某些系统使用领域驱动的模式解决，其他大多数系统都是架构师根据开发经验进行边界的确定
  • CI/CD的自动化并不完全
  • 构建具有容错能力的弹性架构非常重要

RQ3：微服务迁移过程中的主要技术和组织挑战是什么？

  • 成员的协作在瀑布式专项敏捷开发的时候会产生摩擦
  • 架构的转变意味着团队的转变，开发人员会担心自己在新的开发团队中的所处位置

代码克隆

TECCD: A Tree Embedding Approach for Code Clone Detection

ICSE 2019

Main Idea

  • 本文主要研究检测代码克隆问题，本文首先使用树嵌入获得AST中每个中间节点的节点向量，然后使用轻量的方法树的节点向量组合成一个树向量，最后使用欧拉距离来检测树向量之间的相似性来检测代码克隆

背景知识

代码克隆

  • 定义：代码克隆通常是指复制粘贴代码，并进行修改进行使用

  • 分类：代码克隆基于对代码的更改程度被分为四类：

    • Type-1：去除空格，空行，注释等，其他完全相同
    • Type-2：除了对一些重命名操作，其他完全相同
    • Type-3：代码片段被修改，例如添加或者删除代码，以及对部分代码进行重排序
    • Type-4：语义相同，但是句法不相似

  • 检测代码克隆的方法被分为5类：

    • 基于文本的
    • 基于token的
    • 基于树的
    • 基于图的
    • 基于度量的方法

  • Type-3比其他类型的克隆都多，本文主要检测第三种，目前检测这种类型的方法主要是基于token和tree的，基于token的方法对于大量的修改表现较差，基于tree的在大规模检测中效率不足，基于机器学习的方法针对文本大量修改，结构相似的效果较差，因此本文将AST转换为向量，以此进一步检测相似性来检测代码克隆

AST可以捕捉程序结构，这样对于大量修改代码文本，代码结构基本不见的克隆，AST仍能够保留结构信息，然后通过树嵌入获得AST的向量表达，使用向量的相似性来获得AST的相似性，即代码克隆

主要方法

数据预处理

  • 首先使用ANTLR为训练语料中的每个方法生成AST
  • 过滤掉stop node(类似stop word，是冗余的)
  • 在每一个AST上使用random walk，获得每个AST上的节点序列集合(这个序列补货了每个AST的结构信息)
  • 使用节点序列训练一个skip-gram model，生成一个节点向量字典(节点名称是节点向量的索引)

生成树节点集合

  • source code作为输入，ANTLR将源代码转换为AST
  • 使用DFS遍历AST，获得每个AST的树节点，在遍历过程中删除stop node

生成树向量

  • 类似于句子->向量的方法获得AST->向量
  • 不保留AST节点的顺序信息

代码克隆检测

  • 使用LSH来提高向量的比较效率

  • 对于Type-1,Type-2结构基本相同，所以AST基本相同，AST向量的距离基本为0

  • 对于Type-3设置距离阈值为0.02，这个使用数据集手动检查，最后确定这个阈值

  • 但是在检查过程中，添加if-else结构中间节点数量大，因此可以使用可调整的阈值，根据添加代码的比例来调整阈值

借鉴

  • 词嵌入->树嵌入
  • 工具：https://github.com/YangLin-George/TECCD.

代码变更

Aiding Code Change Understanding with Semantic Change Impact Analysis

ICSE 2019

Main Idea

  • 代码审查(code review)主要是开发者手动检查代码变更部分，并理解代码变更的行为，但是由于代码变更的影响区域会超过语法上变更的区域，变更影响分析可以提供所有收到影响变更的部分，本文提出了一种基于语义分析代码变更影响的区域的技术(变更影响分析)

背景

  • 代码审查：开发者需要理解每次提交对于程序的影响
  • 变更影响分析：
    • 静态：主要用来检测程序行为的潜在变化，使用程序切片来计算变化所涉及的语句的控制和数据的依赖性

主要方法

调研了传统的影响分析工具的有效性

  • 传统影响分析工具效果不明显，其中含有噪音：
    • 误报率高:例如静态分析对更名这种操作，认为产生变更，但是实际上对于值并没有更改，使用AST diff解决
    • 不提供代码变更与程序行为变更之间的语义关系

定义了四种新的语义变化影响关系

  • 使用AST diff减少误报
  • 开发基于语义的变更影响的分析

实现了一个分析工具

对新的分析工具进行评价

  • 最后将工具集成到插件中

软件配置

Performance-Influence Model for Highly Configurable Software with Fourier Learning and Lasso Regression

Main Idea

  • 对于可配置的系统的操作和维护中，需要了解不同配置下系统的性能，以及影响系统性能的配置项，本文提出的模型可以使用测量性能值的小样本生成性能测量模型，其可以预测新配置下的系统性能，解释各个特性及其相互作用对软件性能的影响

背景

  • 由于可配置项太多，这就导致组合数量过多，因此需要使用一小部分抽样的样本来估计系统的性能

主要方法(PerLasso)

  • 使用测量的性能数据的小样本来模拟配置选项及其交互对高度可配置软件系统的性能的影响，基于傅立叶学习和 Lasso 回归，并结合了一种新颖的降维算法。

  • 使用 PerLasso 构建的模型有助于了解影响性能的配置选项，并可用于准确预测新配置的系统性能，我们的实验结果证实了所提出方法的有效性。 与现有的性能影响模型相比，我们的模型具有更高或相当的预测精度。

#ifdef Directives and Program Comprehension: The Dilemma between Correctness and Preference(repeat)

在本文中，我们报告了一项大规模实证研究的结果，其中我们分析了重构 CPP 指令如何影响程序理解。 为此，我们创建了一个在线研究，其中包含五个带有特别气味的 CPP 指令的真实代码示例。

其中四个示例被重构以降低指令的复杂性，我们部署了我们的研究，使每个参与者处理两个原始示例和两个重构示例。 第一个示例对所有参与者都是相同的，这使我们能够确定参与者群体在编程经验、年龄和其他个人因素方面具有可比性。 对于每个示例，我们的参与者解决了两个程序理解任务，并提供了他们对代码质量的意见，将客观测量与主观偏好相结合。 我们在开源开发人员中分发了我们的研究，并收到了 521 份回复。 据我们所知，这是第一个结合客观和主观观点的大规模研究，从而大大扩展了之前的研究。 我们得出以下见解：

关于正确性，我们的参与者未能在重构代码上表现得更好，尽管先前的证据表明该代码不应该有坏味道。 事实上，我们的参与者表现稍差。

与原始示例相比，我们的参与者更喜欢在重构示例中使用 CPP 指令的方式。

我们的结果与之前关于重构CPP 指令有用性的发现相互矛盾。

重构CPP 指令需要权衡底层代码的质量，这意味着重构臭指令可能会导致整体质量的净下降。

我们的结果表明了一个在以前的工作中没有报道过的令人惊讶的困境。 这种困境挑战了重构某些 C P P 使用模式（例如无纪律的指令）以提高程序理解的既定信念。

A Method for Finding Missing Unit Tests

Main Idea

  • 测试套件中会缺少某些测试，目前可以通过覆盖率标准来检测缺失测试，这种方法只能指出缺少什么测试，不能提供如何补全这些测试的方法。本文基于相似的代码往往测试方法相同，因此可以通过识别相似的代码是否经过了相似的测试来确定是否缺少测试，然后利用其相似代码测试方法来提出如何编写缺失测试的代码。

背景知识

  • 使用突变测试来寻找缺失测试具有一些缺点，首先需要花费大量的人力和物力，其次不能完全保证没有遗漏的寻找，最后其不能为开发人员提供如何补全缺失测试的方法。
  • 本文通过检测相似代码是否有相似的测试来检测是否缺失测试，例如对于Java中Collection中的add接口，其每个实现目的都是相似的，所以其测试方法也是相似的。因此，我们可以将实现add接口的所有方法的测试进行比较，如果其他实现都有一个测试但是其中一个没有，这说明：1)没有相应测试的实现缺少测试;2)其确实的测试与这个测试目的相同;3)开发者可以参照其他实现的测试方法来编写这个实现的测试方法。

主要研究方法

  • 寻找siblings中的相关方法
  • 总结siblings中测试的共同目的
  • 确定缺失的测试

Improving Automated GUI Exploration of Android Apps via Static Dependency Analysis

Main Idea

  • 为了自动化 GUI 探索，现有技术通常尝试通过特定策略访问尽可能多的 GUI 页面，例如随机（如 Monkey）或启发式（如 Stoat、A3E）。 然而，它们的有效性仍然不清楚，而且还没有得到充分的探索。
  • 为此，我们在本文中进行了第一项研究，通过仔细分析来自一组真实世界的开源应用程序的覆盖率报告来了解和描述它们的局限性。 通过这项研究，我们确定了由于在探索过程中缺乏依赖知识而导致的三个关键限制，即小部件-页面依赖、小部件-小部件依赖和系统-事件依赖。
  • 为了克服这些问题，我们引入了基于依赖的探索，这是一种利用静态依赖分析来有效提高 GUI 探索性能的自动化方法。 给定一个应用程序，我们的方法首先构建一个 GUI 页面转换模型，该模型捕获 GUI 小部件之间的依赖关系，然后在深度优先遍历期间指导 GUI 探索。 我们将我们的方法实现为一个名为 GESDA 的工具，并在 70 个开源 Android 应用程序上对其进行了评估。 结果表明 GESDA 优于现有最先进的 GUI 探索技术，即 Monkey 和 Stoat。 此外，由于依赖知识的好处，GESDA 发现了 4 个应用程序中的 4 个以前未知的崩溃作为 GUI 探索的副产品，而 Monkey 和 Stoat 没有发现它们。

背景知识

  • Android应用以UI为中心，页面通过页面上的组件进行转换。因此可以通过GUI的探索来发现运行时错误，以及安全分析等领域。
  • 自动化GUI探索的目的是为了完全探索所有交互方式以及所有底层逻辑代码，目前已有随机策略和启发式策略来探索GUI。
  • 本文主要通过捕获页面之间的依赖关系，并引导基于深度优先的动态探索。

主要研究方法

  • 节点为页面程序，边表示页面的转换
    • (1) 具有一个或多个事件句柄的widget是widget-page依赖的被依赖的UI元素，应该被完全识别。
    • (2) 状态影响回调控制逻辑的widget是widget-widget依赖的被依赖元素，应该被识别并与包含回调的widget关联。
    • (3)具体的生命周期回调，作为系统事件依赖的触发对象，应该在页面中被识别和预提取。

  • Potential Employees：获取潜在的员工

  • Limited Resources：更多的人参与意味着免精力的获取

  • Business Opportunities：获得潜在的商业合作机会
