TECCD: A Tree Embedding Approach for Code Clone Detection

ICSME 2019

Main Idea

  • 本文主要研究检测代码克隆问题，本文首先使用树嵌入获得AST中每个中间节点的节点向量，然后使用轻量的方法树的节点向量组合成一个树向量，最后使用欧拉距离来检测树向量之间的相似性来检测代码克隆

背景知识

代码克隆

  • 定义：代码克隆通常是指复制粘贴代码，并进行修改进行使用

  • 分类：代码克隆基于对代码的更改程度被分为四类：

    • Type-1：去除空格，空行，注释等，其他完全相同
    • Type-2：除了对一些重命名操作，其他完全相同
    • Type-3：代码片段被修改，例如添加或者删除代码，以及对部分代码进行重排序
    • Type-4：语义相同，但是句法不相似

  • 检测代码克隆的方法被分为5类：

    • 基于文本的
    • 基于token的
    • 基于树的
    • 基于图的
    • 基于度量的方法

  • Type-3比其他类型的克隆都多，本文主要检测第三种，目前检测这种类型的方法主要是基于token和tree的，基于token的方法对于大量的修改表现较差，基于tree的在大规模检测中效率不足，基于机器学习的方法针对文本大量修改，结构相似的效果较差，因此本文将AST转换为向量，以此进一步检测相似性来检测代码克隆

AST可以捕捉程序结构，这样对于大量修改代码文本，代码结构基本不见的克隆，AST仍能够保留结构信息，然后通过树嵌入获得AST的向量表达，使用向量的相似性来获得AST的相似性，即代码克隆

主要方法

数据预处理

  • 首先使用ANTLR为训练语料中的每个方法生成AST
  • 过滤掉stop node(类似stop word，是冗余的)
  • 在每一个AST上使用random walk，获得每个AST上的节点序列集合(这个序列补货了每个AST的结构信息)
  • 使用节点序列训练一个skip-gram model，生成一个节点向量字典(节点名称是节点向量的索引)

生成树节点集合

  • source code作为输入，ANTLR将源代码转换为AST
  • 使用DFS遍历AST，获得每个AST的树节点，在遍历过程中删除stop node

生成树向量

  • 类似于句子->向量的方法获得AST->向量
  • 不保留AST节点的顺序信息

代码克隆检测

  • 使用LSH来提高向量的比较效率

  • 对于Type-1,Type-2结构基本相同，所以AST基本相同，AST向量的距离基本为0

  • 对于Type-3设置距离阈值为0.02，这个使用数据集手动检查，最后确定这个阈值

  • 但是在检查过程中，添加if-else结构中间节点数量大，因此可以使用可调整的阈值，根据添加代码的比例来调整阈值

借鉴

  • 词嵌入->树嵌入
  • 工具：https://github.com/YangLin-George/TECCD.

Predicting Change Propagation between Code Clone Instances by Graph-based Deep Learning

本文主要是针对软件中相似或者相同的克隆代码的修改是否需要传播到其他相似或者相同的代码这一问题进行决策性的研究

Main Idea

  • 本文主要的工作的聚焦于两方向：
    • 首先本文通过实践过程中代码克隆实例之间的变更关系，得出结论某些实例之间是需要一起变更的，而有一些是不需要的
    • 在上述研究的基础上，本文提出了一种基于图卷积神经网络的模型来预测代码克隆实例之间是否需要一起变更

研究背景

  • 代码克隆指的是相似或者相同的代码片段，在实际的项目中，这些片段有时需要保持一致性，因此对于软件维护来说判断这些代码片段是否需要保持一致性是非常重要的

  • 传统的 Program Dependency Graph(PDG) 可以获取程序的数据流和控制流，本文的基于图的代码克隆表示方法可以编码变更的代码克隆实例以及 PDG 中代码克隆实例之间的映射关系。

  • 本文提出的模型是基于 R-GCN 设计的，本文使用的数据集

Git 的三个对象：commit, tree, blob

代码克隆实例一致性传播的研究

  • 数据集：Maven, Ant, DBeaver ,Tomcat, Camel

Git 仓库中代码克隆的检测

  • 在 Git 系统中，对源代码文件的任何更改都会生成一个blob对象。我们对这些 blob 对象进行预处理以生成快照文件，每个快照文件可能被多个相邻的修订版引用。检测这些快照文件中的克隆可确保我们不会遗漏对源代码的任何更改。
  • 代码克隆检测工具：SAGA 基于Token 的代码克隆检测工具
  • 构建代码克隆实例的关系

通过收集数据得到的一些Findings

  • 匹配变化和不匹配变化的百分比在不同的项目中有所不同，表明预测传播需求是一项艰巨的任务。 22.6%-33.7% 的所有克隆系谱都经历了匹配和不匹配的更改，这表明需要考虑每个更改集而不是整个克隆的传播要求。

  • 延迟传播是用于预测代码克隆实例之间的更改传播的不可忽略的数据源，因为在某些软件项目中存在很大比例的延迟传播更改。

  • 克隆对系谱中匹配和不匹配的更改在源代码的文本、结构和上下文中表现出独特的特征。然而，这些特征不容易被启发式规则捕获，这激发了我们的研究，以训练基于深度学习的模型，以更细粒度地预测代码克隆更改传播。
